📋 COMMANDE CLIENT (BC)
🔴 AS-IS : WhatsApp / téléphone → ressaisie manuelle par l'ADV dans un tableau Excel. Risque d'erreur de saisie. Pas de validation prix.
⚙️ Odoo standard : Devis structuré → BC avec lignes produits, prix, UdM, remises. Workflow : Devis → Confirmé → En cours de livraison.
🧩 MVP PARABOX : Contrôle crédit automatique au moment de la confirmation. Blocage si encours + montant BC > limite. Dérogation ADV traçable.
🚚 BON DE LIVRAISON (BL)
🔴 AS-IS : BL papier établi par le chauffeur. Souvent incomplet. Non signé ou signé sans valeur légale. Écarts non documentés.
⚙️ Odoo standard : stock.picking avec move.lines. Validation = quantités done. Lot / N° série obligatoire si tracking='lot'. Backorder automatique.
🧩 MVP PARABOX : Module parabox_sign : OTP 6 chiffres → canvas BIC → GPS → PDF signé → Hash SHA-256 stocké. Preuve non répudiable.
🧾 FACTURATION
🔴 AS-IS : Facturation sur quantité commandée (BC). Si livraison partielle → client conteste → litige. ~3 MDH/an non recouvrés.
⚙️ Odoo standard : Politique de facturation = "Quantités livrées". Facture créée depuis le BC mais montants calculés sur les BL validés.
🧩 MVP PARABOX : Configuration appliquée : sale_default_invoice_policy = 'delivery'. Facturation automatiquement sur livré réel. Backordes non facturés.
💰 PAIEMENT
🔴 AS-IS : Cash, chèques, traites suivis dans Excel. Lettrage manuel. Chèques rejetés non suivis systématiquement.
⚙️ Odoo standard : Paiement simple sur facture. Mode unique par transaction. Suivi basique.
🧩 MVP PARABOX : Module parabox_encaissement : plan multi-lignes (cash + chèque + traite + virement). Statut : En attente → Partiel → Soldé. Gestion rejets.

⚠️ Zones Grises & Gaps Identifiés

Ces gaps ont été identifiés lors de l'analyse PARABOX. Chaque carte montre l'état actuel (AS-IS), ce qu'Odoo standard apporte, et ce que le MVP PARABOX MATCH résout.

🚨 Carte 1 — Commande informelle (WhatsApp → Excel)
🔴 AS-IS : La commande arrive par WhatsApp ou téléphone. L'ADV ressaisit dans Excel. Risque d'oubli, erreur de produit, doublon.
⚙️ Odoo standard : Devis structuré avec liste produits, prix, UdM. Validation et confirmation traçable. Historique client.
🧩 MVP PARABOX : Contrôle crédit à la confirmation (parabox_credit_control). Dérogation ADV loggée. Alerte si encours dépasse limite.
📦 Carte 2 — Boîte noire logistique (4 états non tracés)
🔴 AS-IS : Commandé ≠ Préparé ≠ Chargé ≠ Livré réel. Chaque étape est une boîte noire. Écarts non documentés.
⚙️ Odoo standard : stock.picking avec état (Confirmed → Assigned → Done). Move.lines par produit. Backorder si partiel.
🧩 MVP PARABOX : Module parabox_logistics_tracking : 4 états (Commandé / Préparé / Chargé / Livré). Calcul écart auto. Suivi substitutions. Taux de service par BL.
✍️ Carte 3 — BL papier sans valeur légale
🔴 AS-IS : BL imprimé, signé manuellement par le client. Le livreur perd la feuille, le client conteste sa signature, impossible à prouver en justice.
⚙️ Odoo standard : Bon de livraison numérique avec QR code. Envoi PDF par email. Pas de signature électronique native en Community.
🧩 MVP PARABOX : parabox_sign : OTP 6 chiffres (SHA-256) + canvas BIC + GPS + IP + timestamp → PDF généré + hash SHA-256 stocké → Détection fraude intégrée.
💸 Carte 4 — Paiements multiples non suivis
🔴 AS-IS : Un client paie 2000 cash + 1800 par chèque + 1000 par traite sur une même facture. Suivi dans Excel. Lettrage manuel. Chèques rejetés non tracés.
⚙️ Odoo standard : Paiement simple : un mode, un montant, une date. Plusieurs paiements possibles mais sans plan structuré.
🧩 MVP PARABOX : parabox_encaissement : plan multi-lignes avec modes (cash, chèque, traite, virement), dates d'échéance, statuts (Reçu, Encaissé, Rejeté). Solde auto calculé.
🏷️ Carte 5 — Double étiquetage produit (codes multiples)
🔴 AS-IS : Le même produit a un code PARABOX, un code fournisseur différent, un code revendeur différent, parfois un EAN. Correspondances dans la tête des équipes.
⚙️ Odoo standard : Référence interne (default_code) + EAN sur product.template. Un seul code fournisseur possible.
🧩 MVP PARABOX : parabox_product_alias : table de correspondance code PARABOX / fournisseur / revendeur / EAN. Onglet dédié sur la fiche produit. Filtrable par fournisseur.

👁 Ce qui reste humain — Limites du système

Odoo + les modules MATCH n'éliminent pas le jugement humain. Voici ce qui reste de la responsabilité des équipes.

🤝
Négociation commerciale
Les conditions tarifaires, remises exceptionnelles et gestes commerciaux restent à la discrétion du commercial et de la direction.
🔍
Contrôle qualité physique
Vérification des produits avant livraison (dates, état, conformité). Odoo trace les lots mais ne vérifie pas l'état physique.
📞
Relation client
La résolution d'un litige nécessite un contact humain. Le système fournit les preuves, mais la médiation reste manuelle.
⚖️
Décision de dérogation crédit
L'ADV évalue la situation du client avant d'accorder une dérogation. Odoo fournit les chiffres, mais la décision est humaine.
🧾
Gestion des traites
L'encaissement physique des traites et chèques reste un acte bancaire humain. Odoo trace mais ne présente pas les chèques à la banque.
📱
Mode dégradé terrain
Si le client n'a pas reçu l'OTP (pas de réseau, email incorrect), le livreur doit décider d'utiliser le mode dégradé. Responsabilité humaine.

🧾 Preuves & Auditabilité

🔐 SIGNATURE ÉLECTRONIQUE — Niveau de preuve
✅ Ce qui est prouvable :
  • Le client a reçu l'OTP sur son email (log d'envoi)
  • L'OTP a été saisi correctement (otp_verified = True)
  • La signature a été tracée sur ce device (IP + User-Agent + timestamp)
  • La position GPS au moment de la signature (si autorisée)
  • Le PDF généré n'a pas été modifié depuis (hash SHA-256)
  • Le livreur ayant déclenché la signature (livreur_id)
⚠️ Limite légale : La signature BIC sur canvas ne constitue pas une signature électronique qualifiée au sens de la loi 53-05 (Maroc) ou eIDAS (Europe). Elle constitue cependant une preuve de bonne foi et un commencement de preuve par écrit suffisant pour résoudre la grande majorité des litiges commerciaux.
📋 TRAÇABILITÉ COMPLÈTE
Pour chaque transaction, Odoo trace :
  • Qui a créé / modifié / validé chaque document (create_uid, write_uid)
  • Quand (create_date, write_date)
  • Depuis quelle IP (pour les actions web)
  • Chatter : historique complet des modifications sur chaque enregistrement
🔍 DÉTECTION FRAUDE PDF
🧩 Mécanisme SHA-256 :
  1. Lors de la signature, le PDF est généré par ReportLab avec la signature incrustée
  2. Le hash SHA-256 du PDF est calculé et stocké dans parabox.sign.request.pdf_hash
  3. À tout moment, le bouton "🔒 Vérifier intégrité PDF" recalcule le hash
  4. Si hash_actuel ≠ hash_stocké → alerte "FRAUDE DÉTECTÉE" en rouge
  5. Toute modification du PDF (même d'un seul pixel) change le hash
